## 1. 배경 및 문제 정의

2027년 초 이직 지원(타깃: 네이버, 카카오, 우아한형제들, 당근, 라인 등) 시점에 채용담당자·면접관에게 제시할 기술 브랜딩 자산이 필요하다.

기존 github.io 블로그의 실패 원인 진단:

- **핵심 원인은 플랫폼이 아니라 글감의 질**이었다. UI/UX 불만은 부차적.
- 따라서 이번 리뉴얼의 진짜 차별점은 예쁜 블로그가 아니라, **좋은 글감이 발행의 관문이 되는 워크플로우**다.

기존 repo에서 계속 진행하되, 기존 글(md)은 아카이빙 처리하고(노출 여부는 추후 결정, 논블로킹) 신규 프로젝트로 전면 재구축한다.

## 2. 목표 및 성공 기준

우선순위 순:

1. **이력서에 링크 걸 수 있는 대표 딥다이브 글 3~5개 확보** — 데드라인: 2027년 2월 지원 시작 시점 (제1 성공 기준)
2. 면접 대화 소재가 되는 실무 문제해결 기록 축적
3. 블로그 제작 과정 자체를 글감으로 활용 (과하지 않은 선의 직접 구현)

발행 전략: 주기 강제 없음. **양이 아니라 편당 밀도로 승부.** 습관 형성 장치(스트릭, 알림 등)는 만들지 않는다.

## 3. 독자 정의

- **1차 독자: 채용담당자·면접관.** 대표글의 품질 기준점은 항상 이들.
- 부수 독자: 미래의 나, 동료/주니어 개발자, 검색 유입자.

## 4. 비목표 (Non-goals)

- 수익화 구현 — 목표이되 논블로킹. 단, SEO 세팅은 미래 수익화의 기반 공사로 Must에 포함(개발 막바지 작업).
- 구독자·검색 유입 성장 캠페인 (있으면 좋지만 성공 기준 아님)
- 기존 글 즉시 재노출 (아카이빙 후 추후 결정)
- 글 수정/작성 UI — **블로그·워크벤치 모두 read-only.** 모든 쓰기 작업은 repo 직접 푸시.
- 노션 연동 — 스코프에서 완전 제외. 필요해지면 별도 논의.
- 글감 추천 엔진 — 기존에 구축한 Claude Code 에이전트 시스템이 담당. 블로그는 그 시스템의 출력(점수)을 소비만 한다.

### 안티골 (반복하지 말아야 할 실패 패턴)

- 글감 없이 형식만 갖춘 글 양산 (기존 블로그의 실패 원인)
- 초기부터 DB 등 무거운 인프라 도입 — 파일 기반 유지

## 5. 핵심 워크플로우

```
[draft 저장소]                         [블로그 repo]
거친 draft md 축적          →    발행용 md 푸시     →   블로그 노출
+ 포트폴리오 적합도 점수
       ↑                              ↑
Claude Code 에이전트가          채은이 직접 다듬어
글감 추천/점수 산정              publish 상태로 완성
       ↕
[Draft 워크벤치] (블로그 내 비공개 영역)
점수순/최신순으로 draft 열람, split 미리보기
```

1. 거친 draft를 별도 코드 저장소에 push (기존 점수 시스템이 문서별 포트폴리오 적합도 점수 부여)
2. 발행 의사가 생기면 워크벤치에서 점수순으로 draft 열람, 글감 선택
3. 직접 다듬어 publish 가능한 md로 완성 → 블로그 repo에 푸시
4. 자동 빌드·배포로 노출

## 6. 요구사항 (MoSCoW 확정)

요구사항 ID는 도메인 접두어 + 번호 (BLOG: 블로그 본체 / WB: Draft 워크벤치 / SEO: 검색 최적화). 우선순위는 ID가 아니라 소속 섹션(MoSCoW)이 결정한다.

### Must

| ID     | 항목             | 설명                                                                                              |
| ------ | ---------------- | ------------------------------------------------------------------------------------------------- |
| BLOG-1 | 블로그 코어      | 글 목록 + 글 상세 (md 렌더링, 코드 하이라이팅) + GitHub Pages 배포                                |
| BLOG-2 | 기존 글 아카이빙 | 기존 md를 `/archive` 디렉토리로 이동, 빌드에서 제외 (비노출 보존). 신규 글 축적 후 노출 전환 검토 |
| BLOG-3 | 수동 발행 플로우 | 발행용 md를 repo에 푸시하면 노출되는 파이프라인                                                   |
| SEO-1  | SEO 기본기       | 메타태그, OG, sitemap, RSS. **개발 막바지 작업**                                                  |

### Should

| ID     | 항목                    | 설명                                                                                                                                                                        |
| ------ | ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| WB-1   | Draft 워크벤치          | dev-journal 원장(portfolio-score-ledger.json) 기반 글감 후보(run/chain) 리스트 + 점수 표시, 클릭 시 split view로 md 미리보기, 점수순/최신순 정렬. **read-only**. 상세는 9장 |
| WB-2   | 워크벤치 접근 제어      | 본인 GitHub 인증(fine-grained PAT) 시에만 접근. **draft 데이터는 런타임 GitHub API 조회 — 빌드 산출물에 절대 미포함** (보안의 실체는 private repo 권한)                     |
| BLOG-4 | 대표글 큐레이션 + About | 채용담당자 랜딩 시 대표글 3~5개가 먼저 보이는 구조 + 이력 요약                                                                                                              |
| BLOG-5 | 메타데이터 체계         | frontmatter 스키마: 태그, 시리즈, 작성일, description, 대표글 여부, draft/published 상태. BLOG-4·SEO-1의 기반이므로 스키마는 초반에 확정                                    |

### Could

- OAuth 로그인 (PAT 방식의 업그레이드, 서버 도입 시 자연 획득)
- 빌딩 로그 시리즈 — 이 블로그 제작 과정의 기술 결정을 글로 기록 (기능이 아닌 콘텐츠 전략. 첫 대표글 후보 산지)
- 다크모드, 검색 등 UX 개선류

### Won't (이번 스코프 제외)

글 수정/작성 UI, 노션 연동, 댓글, 구독/뉴스레터, 수익화 구현, 방문자 분석 대시보드

### static export 제약 (명시적 인지 사항)

GitHub Pages 운영 기간 동안 `output: 'export'` 모드 → **API Routes, SSR, ISR, next/image 최적화 사용 불가.** 순수 정적 사이트로 동작한다.

### 향후 확장 경로

- Node 서버 도입 시: 배포처 이전(Vercel 등) 또는 Pages + 별도 API 서버 구성 결정 필요. 이 시점에 OAuth·CRUD가 자연스럽게 열림.
- 수익화: SEO 기반 위에서 별도 검토.

## 9. 워크벤치 데이터 연동 스펙 (WB-1 확정)

- **소스**: `chaeeun037/dev-journal` (private) 저장소의 `momsitter/portfolio-score-ledger.json` 단일 파일. **1회 fetch로 점수·날짜·체인 구조 전부 확보** — 커밋 API 조회 불필요 (N+1 문제 원천 해소).
- **엔티티 모델**: 글감 후보는 파일이 아니라 원장의 두 엔티티 — `runs`(단일 작업, `total` 점수)와 `chains`(서사 묶음, `narrative_value`). 워크벤치는 파일 브라우저가 아닌 **원장 뷰어 + md 미리보기**로 설계한다.
- **표시 규칙** (원장에 정의된 규칙을 UI가 그대로 존중):
  - `current_rubric_version` / `current_chain_rubric_version` 키의 점수만 사용, 구버전 채점은 무시 (원장 히스토리 비대화는 워크벤치에 무영향)
  - `written: true` → "발행됨" 뱃지. 단독 글감 후보에선 제외, 체인 멤버로는 표시 (원장 규칙 동일)
  - `merged: false` / closed → 리스트 제외
  - `provisional: true` 체인 → "추정치" 뱃지
- **정렬**: 점수 내림차순(runs는 `total`, chains는 `narrative_value`) / 최신순(runs는 `created`, chains는 `span.to`)
- **미리보기 경로 매핑**: HARNESS run은 `artifact` 필드 경로 사용. FE run은 pool(`.agents/runs-archive`) + run ID 기반 규칙 — 규칙 확정 필요(미결 1번).
- dims 레이더/막대 시각화(rubric의 `dims_order`·`weights` 활용)는 Could.
